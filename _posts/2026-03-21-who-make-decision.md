    - 경험을 바탕으로 한 1990년대 말의 닷컴 버블기부터 현재까지의 분석.
    - AI를 비롯한 여러 기술 대중화로 인한 미래 예측.

    - "'데이터 스키마'는 서비스가 생산, 활용하는 모든 정보의 도식이며 논리적 설계이다."
    - "'데이터 스키마'는 서비스의 아키텍처와 라이프사이클, 비지니스 모델과 동의어이고 기업 문화, 업무 프로세스, 조직 구조 결정의 핵심 요소중 하나다."
    - 프론트엔드는 사회과학적 특성이 강한 UI/UX 영역으로 축소될 것이고 나머지 영역의 상당부분이 백엔드 영역으로 재분류될 것이다.
    - 백엔드는 프론트엔드에 low level의 데이터 접근을 캡슐화 한 high level의 데이터 접근 도구(라이브러리)를 제공하게 될 것이다.
    - 백엔드는 효율적인 데이터 접근을 구현하기 위해 로컬 캐싱, 로깅, 백그라운드 작업, 플랫폼(OS) 기능을 확장 라이브러리 형식으로 제공하게 될 것이다.
    - 클라이언트 하드웨어의 고성능화로 직관적인 MVVM이 클라이언트 애플리케이션의 기본 패턴이 됐다.
    - MVVM에서 View와 ViewModel은 앞으로도 확실한 프론트엔드 영역이지만 Model의 상당부분은 백엔드 영역이 될 것이다.
    - 서버 하드웨어의 고성능화, 클라우드 인프라, 모바일 환경이 갖춰져서 서버 애플리케이션 개발자가 아키텍트의 주류가 됐다.
    - AI, 네트워크 API 정의 언어와 컴파일러, 멀티플랫폼 프로그래밍 언어의 보급으로 서버 애플리케이션 개발자가 클라이언트용 라이브러리를 개발하기 쉬워졌다.

# 누가 아키텍처를 결정하는가?

미래의 아키텍트는 어떤 스킬셋을 요구할 것인가?

## 요약(TL;DR)

1. 온라인 서비스의 기술 스택에서 아키텍트의 출신은 UI에서 먼 쪽에서 사용자 쪽으로 이동해왔다.
2. 지금은 하드웨어 성능 향상과 그 성능을 활용하는 프레임워크의 보급으로 서버 애플리케이션 개발자 출신의 아키텍트가 많다.
3. 앞으로 아키텍트는 백엔드(서버 애플리케이션과 클라이언트 모델 레이어 일부) 개발자 출신이 될 것이다.

## 아키텍처 결정 권력의 이동

- **전제 정의**
  - 아키텍트 = 서비스 아키텍처를 결정하는 책임자
  - 기준: 기업이 운영하는 B2C 온라인 서비스 (B2B·비영리도 크게 다르지 않음)
  - 논외: 온라인 게임 (핵심 가치가 다름)
  - 패턴: ROI가 높거나, 미션크리티컬하거나, 법적 책임이 있는 기술 스택 전문가가 아키텍트가 됨
- **시기별 변화**
  - ~1990년대 후반: 메인프레임 시대(메인프레임 + 터미널) → 시스템 관리자
  - 1990년대 후반~2000년대 후반: 닷컴 시대(x86 서버, 가상 서버 + PC) → 미들웨어(WAS/ESB) 개발자, DBA
  - 2010년대 초반~현재: 모바일 시대(클라우드 인프라 + 스마트폰) → 서버 애플리케이션 개발자

## 현재 상황

- **용어 정리**: frontend/backend는 시기와 환경에 따라 의미가 다름; 물리적 서버-클라이언트 구조와 논리적 백엔드-프론트엔드 구분이 반드시 일치하지 않음
- **배경**: 하드웨어 고성능화 + 클라우드 인프라 + 모바일 환경 → 인프라·미들웨어 ROI 하락
- **UI/UX의 성격**: 사회과학적 특성이 강해 시행착오 기반; 백엔드 대비 수정 비용 낮음
- **백엔드의 위상**: 데이터 스키마·비즈니스 로직·법적 책임 → 신중한 설계 필요, 수정 비용 높음
- **데이터 스키마 정의**: 서비스가 생산·활용하는 모든 정보의 도식이자 논리적 설계
- **결과**: 클라이언트 고성능화로 MVVM이 기본 패턴이 됐고, 서버 애플리케이션 개발자가 전체 아키텍처 결정권을 가짐

## 앞으로의 변화

- **프론트엔드의 재정의**
  - 프론트엔드 → UI/UX(사회과학 영역)로 축소
  - MVVM에서 View·ViewModel은 프론트엔드, Model의 상당 부분은 백엔드로 재분류
- **백엔드 영역의 확장**
  - 백엔드가 프론트엔드에 low-level 데이터 접근을 캡슐화한 high-level 라이브러리 제공
  - 로컬 캐싱, 로깅, 백그라운드 작업, 플랫폼(OS) 기능도 확장 라이브러리 형태로 백엔드가 담당 가능
- **가능하게 하는 기술**
  - API 정의 언어·컴파일러(gRPC, GraphQL): 서버 개발자가 클라이언트 라이브러리 자동 생성
  - 멀티플랫폼 언어(Kotlin Multiplatform, Dart 등): 서버 개발자의 클라이언트 라이브러리 개발 난이도 낮춤
  - AI 코드 생성 도구: UI 개발 비용 하락, 영속화 데이터 관리는 사람의 검토 영역으로 잔존
- **귀결**
  - 데이터 스키마 = 서비스 아키텍처·라이프사이클·비즈니스 모델과 동의어
  - 기업 전체의 데이터 스키마를 설계·진화시킬 수 있는 백엔드 개발자가 아키텍트가 될 것
