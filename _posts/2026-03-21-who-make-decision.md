    - 경험을 바탕으로 한 1990년대 말의 닷컴 버블기부터 현재까지의 분석.
    - AI를 비롯한 여러 기술 대중화로 인한 미래 예측.
    - 잘 알려진 용어이지만 시기와 환경에 따라 서로 다른 뜻인 경우가 많다.
    - 백엔드-프론트엔드의 논리적 구분은 서버-클라이언트의 물리적 구조와 일치해야 할 필연적 이유가 없고 고정된 구분도 아니다.
    - "'데이터 스키마'는 서비스가 생산, 활용하는 모든 정보의 도식이며 논리적 설계이다."
    - "'데이터 스키마'는 서비스의 아키텍처와 라이프사이클, 비지니스 모델과 동의어이고 기업 문화, 업무 프로세스, 조직 구조 결정의 핵심 요소중 하나다."
    - 프론트엔드는 사회과학적 특성이 강한 UI/UX 영역으로 축소될 것이고 나머지 영역의 상당부분이 백엔드 영역으로 재분류될 것이다.
    - 백엔드는 프론트엔드에 low level의 데이터 접근을 캡슐화 한 high level의 데이터 접근 도구(라이브러리)를 제공하게 될 것이다.
    - 백엔드는 효율적인 데이터 접근을 구현하기 위해 로컬 캐싱, 로깅, 백그라운드 작업, 플랫폼(OS) 기능을 확장 라이브러리 형식으로 제공하게 될 것이다.
    - 클라이언트 하드웨어의 고성능화로 직관적인 MVVM이 클라이언트 애플리케이션의 기본 패턴이 됐다.
    - MVVM에서 View와 ViewModel은 앞으로도 확실한 프론트엔드 영역이지만 Model의 상당부분은 백엔드 영역이 될 것이다.
    - 서버 하드웨어의 고성능화, 클라우드 인프라, 모바일 환경이 갖춰져서 서버 애플리케이션 개발자가 아키텍트의 주류가 됐다.
    - AI, 네트워크 API 정의 언어와 컴파일러, 멀티플랫폼 프로그래밍 언어의 보급으로 서버 애플리케이션 개발자가 클라이언트용 라이브러리를 개발하기 쉬워졌다.

# 누가 구조를 결정하는가?

미래의 아키텍트는 어떤 스킬셋을 요구할 것인가?

## 요약(TL;DR)

1. 온라인 서비스의 기술 스택에서 아키텍트의 출신은 UI에서 먼 쪽에서 사용자 쪽으로 이동해왔다.
2. 지금은 하드웨어 성능 향상과 그 성능을 활용하는 프레임워크의 보급으로 서버 애플리케이션 개발자 출신의 아키텍트가 많다.
3. 앞으로 아키텍트는 백엔드(서버 애플리케이션과 클라이언트 모델 레이어 일부) 개발자 출신이 될 것이다.
