---

소프트웨어 설계를 뭐라고 정의할 수 있을까 꽤 오래 고민했었다.

그리고 내 정의는

> 소프트웨어 설계란 독자(개발자)의 맥락을 제어하는 체계.

이다. 이 정의를 이해하려면 소스코드의 정의도 필요하다.

> 소스코드란 프로세서가 작업을 수행할 수 있을 정도로 자세하게 작성한 작업 매뉴얼이다.

라고 소스코드를 정의할 수 있다. 그렇다면

> 개발자란 프로세서용 작업 매뉴얼을 작성하고 관리하는 사람.

라고 개발자를 정의할 수 있다.

따라서 소프트웨어 설계는 매뉴얼 작성, 관리, 독자를 위해 맥락을 제어하는 쳬계이고, 구체적인 제어 대상은 다음과 같다.

- 도메인 지식
  - 비지니스 로직을 어느 정도로 노출할 것인가. 온라인 쇼핑몰이 "주문 확인 -> 결재 확인 -> 재고 확인 -> 배송 확정"의 로직을 가지고 있다면, 어떻게 주문을 확인하고 어떻게 결재를 확인할 것인가 하는 상세 로직을 다른 코드로 분리하면서 숨긴다.
  - "주문 처리"가 전체적으로 "주문 확인 -> 결재 확인 -> 재고 확인 -> 배송 확정" 흐름임을 명확히 하면서 구체적인 "주문 확인" 방법은 맥락에서 제외할 수 있다. 반면 "주문 확인" 방법을 다룰 땐 "주문 처리"를 맥락에서 제외할 수 있다.
- 사용하는 기술 스택.
  - 주문이나 결재 정보가 어디에 저장되어 있는지 숨겨서 구현을 교체할 수 있는 방법으로 인터페이스만 노출하기.
  - DB에 있을 수도 있고, 파일에 있을 수도 있고, 외부 서비스에 있을 수도 있다.
  - "주문 처리"를 작업할 땐 DB 기술 정보를 맥락에서 제외할 수 있고, "주문 확인"을 작업할 때는 DB 기술만 포함하고 외부 서비스를 제외할 수 있다.
- 코드에 접근하고 수정할 권한.
  - 하나의 파일에서 라인 단위로 담당자를 정하는 것 보다는 파일 < 모듈 < 저장소 순으로 권한을 명시적으로 제어하기 쉬워진다.
  - "주문 처리"를 작업할 땐 `public`인 인터페이스만 맥락에 포함하고, `private`/`internal`인 구현은 맥락에서 제외할 수 있다.
  - 저장소 단위로 코드를 관리한다면 저장소 접근 권한으로 개발자의 맥락을 물리적으로 제어할 수 있다. 다른 수단은 개발 프로세스나 다른 도구로 관리할 수 있다.
