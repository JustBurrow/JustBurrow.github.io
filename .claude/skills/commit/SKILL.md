현재 작업 디렉토리의 모든 변경사항(스테이지 여부 무관)을 분석해 성격별로 원자적 커밋으로 나눠 커밋하라.

## 절차

1. `git status`와 `git diff HEAD`(스테이지 포함)로 전체 변경사항을 파악한다.
2. 변경된 파일 각각을 아래 커밋 유형 중 하나로 분류한다.
3. 같은 유형으로 묶인 파일을 하나의 원자적 커밋으로 묶는다.
   - 단, 서로 다른 기능·관심사에 해당하면 같은 유형이라도 커밋을 분리한다.
4. 분류 결과와 예정 커밋 목록을 사용자에게 보여주고 확인을 받는다.
5. 확인 후 각 커밋을 순서대로 실행한다.

## 커밋 유형과 gitmoji

| 유형 | gitmoji | 설명 |
|------|---------|------|
| 새 기능 | ✨ | 새로운 기능 추가 |
| 버그 수정 | 🐛 | 버그 수정 |
| 문서 | 📝 | 문서 추가·수정 |
| 스타일 | 💄 | UI/CSS, 포맷 변경 (로직 무관) |
| 리팩터 | ♻️ | 동작 변경 없는 코드 개선 |
| 성능 | ⚡️ | 성능 개선 |
| 테스트 | ✅ | 테스트 추가·수정 |
| 빌드·의존성 | 📦 | 패키지·빌드 설정 변경 |
| CI/CD | 👷 | CI/CD 설정 변경 |
| 설정 | 🔧 | 설정 파일 변경 |
| 삭제 | 🔥 | 코드·파일 삭제 |
| 이동·이름 변경 | 🚚 | 파일 이동·이름 변경 |
| 초기 커밋 | 🎉 | 프로젝트 시작 |
| 콘텐츠 | 🖊️ | 블로그 포스트 등 콘텐츠 추가·수정 |
| 기타 | 🔨 | 위 유형에 해당하지 않는 변경 |

gitmoji 전체 목록은 https://gitmoji.dev/ 를 참고한다.

## 커밋 메시지 형식

```
{gitmoji} {한줄 요약}

{자세한 내용}
```

- **한줄 요약**: 명령형 현재 시제, 한국어, 50자 이내
- **자세한 내용**: 변경한 이유·맥락 위주. 무엇을 바꿨는지보다 왜 바꿨는지를 설명. 필요 없으면 생략 가능.

## 주의사항

- 실행 전 반드시 계획을 보여주고 사용자 확인을 받는다.
- `--no-verify` 플래그는 사용하지 않는다.
- `.env`, 시크릿, 바이너리 대용량 파일은 커밋에서 제외하고 사용자에게 알린다.
- 신규 파일·수정 파일만 포함하고 `git add -A`는 사용하지 않는다 (의도치 않은 파일 포함 방지).
